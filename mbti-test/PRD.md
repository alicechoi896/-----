# PRD: MBTI 성격 유형 테스트 웹사이트

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 서비스명(가칭) | 나의 MBTI 찾기 (MyMBTI) |
| 한 줄 정의 | 12개 질문에 답하면 16가지 MBTI 유형 중 하나를 알려주고, 결과를 이미지로 저장·공유할 수 있는 웹 테스트 서비스 |
| 핵심 가치 | ① 짧고 부담 없는 테스트(1~2분 내 완료) ② 결과를 예쁘게 저장/공유하고 싶게 만드는 비주얼 ③ "나만 이런 결과인가?" 궁금증을 해소하는 실시간 통계 |
| 플랫폼 | 반응형 웹 (모바일 브라우저 우선 최적화, PC 대응) |
| 로그인/회원가입 | 없음 — 누구나 즉시 접근 가능한 비회원 서비스 |
| 서비스 성격 | 바이럴/공유 지향 캐주얼 콘텐츠 (설문 결과가 SNS에 공유되며 신규 유입을 만드는 구조) |

**가정(Assumption)**: 사용자가 로그인 여부를 명시하지 않았으나, MBTI 테스트류 서비스 특성상 진입장벽을 낮추는 것이 바이럴에 유리하므로 비회원 기반으로 설계함. 회원 기능(테스트 이력 저장 등)은 범위 밖으로 두고, 확장 가능성만 남겨둠.

## 2. 타깃 사용자 분석

| 페르소나 | 특징 | 니즈 |
|---|---|---|
| **A. 재미 추구형 (핵심 타깃)** | 10~20대, SNS를 통해 유입, 친구와 결과를 비교/공유하는 것을 즐김 | 빠른 테스트, 예쁜 결과 이미지, 쉬운 공유 버튼 |
| **B. 자기 이해 탐색형** | 20~30대, 자기계발/커리어 고민 중 유입 | 유형별 강점/약점, 추천 직업 등 신뢰감 있는 콘텐츠 |
| **C. 재방문/비교형** | 이미 자신의 MBTI를 알고 있고, 이 서비스의 결과가 맞는지 검증하러 옴 + 통계 페이지에서 "내 유형이 흔한지" 궁금해함 | 통계 기능, 결과의 설득력(질문 퀄리티) |

**UX 함의**: 세 페르소나 모두 공통적으로 "빠름 + 저부담 + 공유하고 싶은 결과물"을 원함 → 회원가입 없이 즉시 시작, 12문항으로 시간 최소화, 결과 페이지 비주얼에 투자.

## 3. 핵심 기능 목록

| # | 기능 | 상세 요구사항 |
|---|---|---|
| 1 | **12문항 테스트** | E/I, S/N, T/F, J/P 4개 지표를 각 3문항씩 판별. 각 질문은 2지선다(예: "친구들과 있을 때 나는? A. 에너지가 충전된다 / B. 에너지가 소모된다"). 문항 순서는 4개 지표가 골고루 섞이도록 배치(EI→SN→TF→JP 반복 등 특정 지표에 몰리지 않게) |
| 2 | **프로그레스 바** | 화면 상단 고정, 현재 문항/전체 문항(예: "5 / 12") 텍스트 + 채워지는 바 형태로 시각화. 답변 선택 즉시 다음 문항으로 부드럽게 전환(슬라이드 애니메이션) |
| 3 | **이전 문항으로 돌아가기** | 뒤로가기 버튼으로 직전 답변 수정 가능(사용자 신뢰도를 높이기 위한 보조 기능) |
| 4 | **결과 계산 로직** | 4개 지표별로 3문항 중 더 많이 선택된 특성을 채택(다수결). 예: E/I 문항 3개 중 E 2회, I 1회 선택 시 최종 E로 판정. 4개 지표 조합으로 16유형 중 하나 확정 |
| 5 | **결과 페이지** | 4글자 유형(예: INFP) + 유형 애칭(예: "열정적인 중재자") + 유형 설명, 강점 리스트, 약점 리스트, 추천 직업 리스트를 섹션별로 표시. 유형 그룹(분석가 NT/외교관 NF/관리자 SJ/탐험가 SP)별 테마 컬러 적용 |
| 6 | **결과 이미지 저장** | 결과 카드(유형+애칭+대표 특징 요약)를 PNG 이미지로 다운로드 가능 |
| 7 | **결과 공유** | 카카오톡 공유, 링크 복사, X(트위터) 공유 버튼 제공. 공유된 링크를 SNS에서 열면 해당 유형에 맞는 미리보기 카드(제목/이미지)가 노출되도록 Open Graph 메타태그를 결과별로 동적 생성 |
| 8 | **통계 페이지** | 지금까지 테스트를 완료한 전체 사용자 기준, 16개 유형별 비율을 가로 막대그래프로 표시(%, 인원수). "내 유형은 전체의 N%" 문구를 결과 페이지에도 함께 노출해 통계 페이지 유입 유도 |

## 4. 사용자 플로우

**메인 플로우: 랜딩 → 테스트 → 결과 → 공유**

1. 사용자가 랜딩 페이지 접속 → 서비스 소개 카피 + "테스트 시작하기" 버튼 확인
2. "테스트 시작하기" 클릭 → 1번 문항 화면으로 전환, 상단 프로그레스 바 "1 / 12" 표시
3. 2지선다 중 하나를 탭/클릭 → 자동으로 다음 문항으로 슬라이드 전환(별도의 "다음" 버튼 없이 선택=진행)
4. 12번 문항까지 반복. 필요 시 상단 "이전" 버튼으로 되돌아가 수정 가능
5. 12번째 답변 완료 → 로딩(결과 계산) 연출 1~2초 → 결과 페이지로 전환
6. 결과 페이지에서 유형/애칭/특징을 확인, 아래로 스크롤하며 강점→약점→추천 직업 확인
7. 하단 고정 공유 바에서 "이미지 저장" 또는 "카카오톡 공유" 등 선택
8. 결과 페이지 내 "전체 통계 보기" 버튼 클릭 → 통계 페이지로 이동, 자신의 유형이 전체에서 몇 %인지 확인
9. "테스트 다시하기" 버튼으로 처음부터 재시작 가능

**보조 플로우: 공유 링크로 유입된 신규 사용자**

- 친구가 공유한 결과 링크 클릭 → 해당 친구의 결과 요약 페이지(간단 프리뷰) 노출 → "나도 테스트하기" CTA 버튼으로 자연스럽게 신규 테스트 플로우 진입

## 5. UI/UX 요구 사항

### 5.1 전체 톤앤매너
- 모바일 퍼스트, 최대 480px 폭 카드 형태로 중앙 정렬(PC에서도 좁은 폭 유지해 집중도 향상)
- 라운드 카드 + 파스텔 톤 컬러 팔레트, 폰트는 큼직하고 가독성 높은 산세리프
- 다크모드 지원(시스템 설정 자동 감지)

### 5.2 테스트 화면
- 문항 텍스트는 화면 중앙 상단에 크게, 두 선택지는 세로로 쌓인 큼직한 버튼(엄지로 누르기 쉬운 최소 높이 56px) 형태로 배치 — 라디오 버튼 지양
- 프로그레스 바는 화면 최상단에 고정(sticky), 얇은 바 + 숫자 텍스트 병기
- 문항 전환 시 좌→우 슬라이드 + 페이드 애니메이션(200~300ms), 로딩처럼 느껴지지 않게 즉각 반응
- 선택 즉시 버튼에 짧은 하이라이트 피드백(색상 변경) 후 전환

### 5.3 결과 페이지
- 최상단: 큰 4글자 유형 타이포 + 유형 애칭 + 유형 그룹 컬러가 반영된 배경/일러스트 영역
- 이어서 카드 섹션 순서: 유형 요약 → 강점(체크 아이콘 리스트) → 약점(주의 아이콘 리스트) → 추천 직업(태그/칩 형태)
- "이 유형은 전체 응답자의 N%" 문구를 요약 바로 아래 배치해 통계 기능으로 자연스럽게 연결
- 하단에 공유 액션 바를 화면에 고정(sticky footer): [이미지 저장] [카카오톡] [링크 복사] [X 공유] 아이콘 버튼 4개 가로 배치
- 저장용 이미지는 별도의 "공유 카드" 컴포넌트로 디자인(로고/유형/애칭/대표 강점 1~2개만 압축 노출 — 정보 과다 방지)

### 5.4 통계 페이지
- 16개 유형을 가로 막대그래프로, 비율 높은 순 정렬
- 각 막대는 유형 그룹 컬러 적용, 막대 끝에 %와 인원수 병기
- 사용자 본인이 방금 받은 유형은 막대를 강조 표시(테두리/굵게)로 하이라이트

### 5.5 접근성/성능
- 모든 버튼 44px 이상 터치 영역, 명도 대비 WCAG AA 기준 충족
- 이미지 저장/공유 액션은 로딩 상태(스피너)를 명확히 표시해 중복 클릭 방지

## 6. 기술 스택

| 구분 | 선택 | 근거 |
|---|---|---|
| 프레임워크 | **Next.js 16.2.x** (App Router) | 결과 공유 링크가 카카오톡/X 등에서 미리보기 카드로 보이려면 유형별로 다른 Open Graph 메타태그를 서버에서 동적으로 렌더링해야 함 — 순수 SPA(Vite)로는 이 요구사항을 만족하기 어려워 SSR/메타데이터 API를 지원하는 Next.js를 선택 |
| 언어 | TypeScript | 질문/결과/통계 데이터 모델의 타입 안전성 확보 |
| 스타일링 | CSS Modules 또는 Tailwind CSS | 컴포넌트 단위 스타일 캡슐화, 파스텔 팔레트/다크모드 변수 관리 용이 |
| 결과 이미지 생성 | **html-to-image** | DOM 요소(공유 카드)를 그대로 PNG로 변환. html2canvas보다 가볍고 최신 프로젝트에서 더 널리 채택됨 |
| 통계 차트 | **Recharts 3.9.x** | React 선언형 API로 막대그래프 구현이 간단하고, 커스텀 컬러/애니메이션 지원 |
| 데이터베이스 | **Postgres (예: Supabase 또는 Vercel Postgres)** | 통계 기능은 전체 사용자 응답이 누적 집계되어야 하므로 클라이언트 로컬 저장(localStorage)이 아닌 서버 DB 필요. 응답 로그를 단순 테이블로 저장해 집계 쿼리로 통계 산출 |
| API | Next.js Route Handlers (`/app/api/...`) | 결과 제출(POST) 및 통계 조회(GET) 엔드포인트를 프레임워크 내에서 자체 구현, 별도 백엔드 서버 불필요 |
| 배포 | Vercel | Next.js와 통합이 가장 간단하고 무료 티어로 MVP 운영 가능 |

## 7. 데이터 구조

### 7.1 정적 콘텐츠 데이터 (코드 내 상수, DB 불필요)

```ts
// 질문 데이터
interface Question {
  id: number                       // 1~12
  dichotomy: 'EI' | 'SN' | 'TF' | 'JP'
  text: string
  options: [
    { text: string; trait: Trait },
    { text: string; trait: Trait },
  ]
}
type Trait = 'E' | 'I' | 'S' | 'N' | 'T' | 'F' | 'J' | 'P'

// 유형 설명 데이터 (16종)
interface MbtiTypeInfo {
  type: string            // 'INFP'
  nickname: string        // '열정적인 중재자'
  group: 'NT' | 'NF' | 'SJ' | 'SP'
  summary: string
  strengths: string[]
  weaknesses: string[]
  recommendedJobs: string[]
}
```

### 7.2 클라이언트 진행 상태 (세션 내 메모리, 저장 불필요)

```ts
interface TestProgress {
  currentQuestionIndex: number
  answers: { questionId: number; trait: Trait }[]
}
```

### 7.3 서버 DB 스키마 (통계용 최소 로그)

```sql
CREATE TABLE mbti_results (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  mbti_type   CHAR(4) NOT NULL,      -- 예: 'INFP'
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```
- 개인 식별 정보(PII) 없음 — 유형 값과 타임스탬프만 저장해 프라이버시 부담 최소화
- 통계 조회는 `GROUP BY mbti_type`으로 집계(실시간 쿼리 또는 캐싱된 결과 사용)

### 7.4 API 인터페이스

| 엔드포인트 | 메서드 | 설명 |
|---|---|---|
| `/api/results` | POST | 테스트 완료 시 계산된 유형을 `mbti_results`에 1건 기록 |
| `/api/stats` | GET | 유형별 인원수/비율을 집계해 반환 (`[{type, count, percentage}]`) |
| `/result/[type]` | 페이지 | 유형별 동적 OG 메타태그가 적용된 결과/공유 랜딩 페이지 |

## 8. 개발 우선순위 (Phase 단위)

### Phase 1 — MVP (테스트 → 결과 확인)
- 12문항 데이터 및 2지선다 UI
- 문항 진행 로직 + 결과 계산(다수결 로직)
- 결과 페이지: 유형/애칭/특징/강점/약점/추천 직업 정적 렌더링
- 프로그레스 바(기본 버전)

### Phase 2 — UX 완성도
- 문항 전환 애니메이션, 이전 문항으로 돌아가기
- 결과 페이지 비주얼(유형 그룹별 테마 컬러, 레이아웃 다듬기)
- 다크모드, 반응형 최종 점검

### Phase 3 — 공유/저장 기능
- 공유 카드 컴포넌트 + html-to-image 기반 이미지 저장
- 카카오톡/X 공유, 링크 복사
- 유형별 동적 OG 메타태그(`/result/[type]`) 구현

### Phase 4 — 통계 기능
- DB 스키마 구축 및 결과 제출 API(`POST /api/results`)
- 통계 집계 API(`GET /api/stats`) 및 Recharts 기반 통계 페이지
- 결과 페이지에 "전체 응답자의 N%" 실시간 반영
